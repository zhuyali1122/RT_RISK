# 回收率计算公式整理

## 一、基础回收率（收益页 / 现金流）

### 1.1 公式

```
回收率 = 实际回收 ÷ 应回收
```

### 1.2 数据来源

| 项目 | 说明 |
|------|------|
| **实际回收 (collection)** | 当月 `raw_repayment` 汇总：本金 + 利息 + 罚息 + 展期费等 |
| **应回收 (expected_due)** | `raw_loan.repayment_schedule` 中 `due_date` 落在当月的应还本金 + 应还利息之和 |

### 1.3 边界处理

- 若 **应回收 = 0** 且 **实际回收 > 0**：回收率取 **98%**
- 若 **应回收 = 0** 且 **实际回收 = 0**：回收率取 **0**
- 回收率限制在 **[0, 1]** 区间

### 1.4 代码位置

- `kn_revenue.py`：收益规模按月计算 `collection_rate`
- `kn_cashflow.py`：现金流预测使用该回收率

## 二、现金流预测中的回收率

### 2.1 公式

```
预期回收 = (应还本金 + 应还利息) × 回收率
```

按月份汇总：每月 `expected_inflow = (principal + interest) × 回收率`

### 2.2 回收率来源（优先级）

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | 收益规模最新月 | `revenue_data[-1].collection_rate` = 实际回收 ÷ 应回收 |
| 2 | 默认值 | **98%**（无历史数据时） |

### 2.3 数据口径

- **应还本金/利息**：`repayment_schedule` 中 `due_date > 最新数据日` 的未到期应还
- **活跃贷款**：`calc_overdue` 中 `loan_status IN (1, 2)` 的未结清贷款

## 三、预期回款/收益中的综合回收率（Vintage 模型）

### 3.1 公式

```
预期回收率 = P(不违约) × 正常回收率 + P(违约) × 违约回收率
           = survival × 正常回收率 + (1 - survival) × 违约回收率
```

### 3.2 参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| **正常回收率** | 98% | 正常还款时的回收比例 |
| **违约回收率** | 30% | 违约后催收可回收比例 |
| **survival** | ∏(1 - default_mob_i) | 不违约概率，由 Vintage 各 MOB 的 default 率计算 |

### 3.3 预期值计算

```
预期总回款 = Σ [ 合同金额 × 预期回收率 ]
预期总收益 = Σ [ 未来利息 × 预期回收率 ]
```

其中 **合同金额** = 未来应还本金 + 未来应还利息（来自 `repayment_schedule`）

### 3.4 代码位置

- `scripts/expected_repayment_revenue.py`：`NORMAL_RECOVERY_RATE = 0.98`，`DEFAULT_RECOVERY_RATE = 0.30`

## 四、回收报表中的回收档位（d1/d3/d7/d30/d60/d90）

### 4.1 说明

回收报表按 **DPD 档位** 统计实际回收金额，不直接计算“回收率”，而是展示各档位的回收金额。

### 4.2 DPD 档位映射

| DPD 范围 | 档位 |
|----------|------|
| 1–2 | d1 |
| 3–6 | d3 |
| 7–29 | d7 |
| 30–59 | d30 |
| 60–89 | d60 |
| 90+ | d90 |

### 4.3 输出字段

- `d1_recovery`, `d3_recovery`, `d7_recovery`, `d30_recovery`, `d60_recovery`, `d90_recovery`：各档位实际回收金额

## 六、回收报表详细计算逻辑（kn_collection.py）

### 6.1 整体结构

回收报表按 **到期月 (maturity_month)** 汇总，每行包含：
- **到期金额**：该月到期贷款的总应还金额
- **入催**：按 DPD 档位汇总的未偿本金（d0~d90）
- **回收**：按 DPD 档位汇总的实际回收金额（d1~d90）

### 6.2 到期月与到期金额

**数据来源**：`raw_loan` + `repayment_schedule`

**计算步骤**：
1. 每笔 loan 的 **maturity_month**：
   - 优先用 `raw_loan.loan_maturity_date` 的年-月
   - 若无，取 `repayment_schedule` 最后一期 `due_date` 的年-月
2. 每笔 loan 的 **total_due**：`repayment_schedule` 中所有期数的 `principal + interest` 之和
3. 按 maturity_month 汇总：`due_amount = SUM(total_due)`

**注意**：这里汇总的是**整笔贷款的全部应还**，不是单月到期应还。

### 6.3 入催（into_collection）

**数据来源**：`calc_overdue` + `raw_loan`，`stat_date = 报表日期`，`loan_status IN (1, 2)`

**计算步骤**：
1. 每笔 loan 的 maturity_month（同上）
2. 每笔 loan 在 stat_date 的 **dpd** 和 **outstanding_principal**
3. 按 DPD 归入档位：

| DPD | 入催档位 |
|-----|----------|
| 0 | d0 |
| 1–2 | d1 |
| 3–6 | d3 |
| 7–29 | d7 |
| 30–59 | d30 |
| 60–89 | d60 |
| 90+ | d90 |

4. 按 (maturity_month, dpd_bucket) 汇总：`d0_into_collection` = 该月到期且 DPD=0 的 loan 的 outstanding_principal 之和，以此类推。

**含义**：入催 = 该到期月的贷款中，当前处于各 DPD 档位的**未偿本金余额**。

### 6.4 回收（recovery）— 算法详解

#### 6.4.1 输入数据

| 来源 | 字段 | 说明 |
|------|------|------|
| raw_repayment | loan_id, repayment_date, principal_repayment, interest_repayment | 所有历史还款记录（无日期过滤） |
| raw_loan | loan_id → maturity_month | 每笔贷款的到期月 |
| calc_overdue | (loan_id, stat_date) → dpd | 各日快照中每笔贷款的 DPD |

#### 6.4.2 算法流程（伪代码）

```
对 raw_repayment 中每一笔还款 (loan_id, repayment_date, amt):
  1. amt = principal_repayment + interest_repayment
  2. 若 amt <= 0，跳过

  3. 查该 loan 的 maturity_month = loan_to_mm[loan_id]
     若 maturity_month 不在报表的到期月列表中，跳过

  4. 查 repayment_date 当天的 DPD：
     dpd = dpd_map[(loan_id, repayment_date)]
     dpd_map 来自 calc_overdue：key = (loan_id, stat_date)，value = dpd
     ⚠️ 必须 stat_date 与 repayment_date 完全一致才有值

  5. 若 dpd 为 None 或 < 1，跳过（DPD=0 的还款不进入回收统计）

  6. 按 dpd 归入档位：
     dpd 1–2   → bucket = "d1"
     dpd 3–6   → bucket = "d3"
     dpd 7–29  → bucket = "d7"
     dpd 30–59 → bucket = "d30"
     dpd 60–89 → bucket = "d60"
     dpd 90+   → bucket = "d90"

  7. 累加：recovery_by_month[maturity_month][bucket + "_recovery"] += amt
```

#### 6.4.3 关键逻辑说明

**① 为何用 repayment_date 当天的 DPD？**

- 回收统计的是「在逾期 X 天时收回的金额」
- 还款发生当天的 DPD 代表当时所处的逾期阶段
- 例如：DPD=35 时还款 → 归入 d30_recovery（30–59 天档位）

**② 为何 DPD=0 的还款不统计？**

- `_dpd_bucket_recovery` 对 dpd < 1 返回 None，直接跳过
- DPD=0 视为正常还款，不进入「催收回收」的 DPD 档位统计
- 报表中的 d1~d90_recovery 仅统计逾期后的回收

**③ dpd_map 的构建方式**

- 收集 raw_repayment 中所有 repayment_date 出现的 (年, 月)
- 对每个 (年, 月)，查询 `calc_overdue_y{年}m{月}` 的全部 (loan_id, stat_date, dpd)
- 得到 `dpd_map[(loan_id, stat_date)] = dpd`
- 查找时用 `(loan_id, repayment_date)` 作为 key
- **前提**：calc_overdue 中存在 `stat_date = repayment_date` 的快照，否则 dpd 为 None，该笔还款被跳过

**④ 按 maturity_month 汇总的含义**

- 每笔还款按**该笔贷款所属的到期月**归入对应行
- 例如：到期月 2026-03 的贷款，在 DPD 35 时还款 1000 → 计入 2026-03 行的 d30_recovery
- 同一到期月的多笔还款，按 DPD 档位分别累加

#### 6.4.4 示例

| 还款 | loan_id | repayment_date | amt | 该日 DPD | 归入 | maturity_month |
|------|---------|----------------|-----|----------|------|----------------|
| A | L001 | 2026-02-15 | 500 | 5 | d3_recovery | 2026-03 |
| B | L001 | 2026-02-20 | 300 | 10 | d7_recovery | 2026-03 |
| C | L002 | 2026-02-10 | 1000 | 0 | 跳过 | 2026-04 |
| D | L003 | 2026-01-28 | 200 | 45 | d30_recovery | 2026-02 |

结果：2026-03 行 d3_recovery +500，d7_recovery +300；2026-02 行 d30_recovery +200。

### 6.5 数据流示意

```
raw_loan (loan_id, loan_maturity_date, repayment_schedule)
    → maturity_month, due_amount

calc_overdue (stat_date=报表日, loan_id, dpd, outstanding_principal)
    → 入催：按 maturity_month + DPD 档位汇总 outstanding_principal

raw_repayment (loan_id, repayment_date, principal_repayment, interest_repayment)
    + calc_overdue (stat_date=repayment_date, loan_id, dpd)
    → 回收：按 maturity_month + 还款日 DPD 档位汇总 还款金额
```

### 6.6 注意事项

1. **回收的 DPD 取还款日**：一笔还款归入哪个档位，取决于**还款发生当天**该 loan 的 DPD，而非报表日。
2. **calc_overdue 需有对应日快照**：若 repayment_date 在 calc_overdue 中无对应 stat_date，该笔回收无法归入档位。
3. **到期月口径**：maturity_month 为整笔贷款的到期月，不是单期还款的到期月。

## 五、汇总对照表

| 场景 | 公式 | 回收率来源 |
|------|------|------------|
| 收益页 | 回收率 = 实际回收 ÷ 应回收 | 当月实际数据 |
| 现金流预测 | 预期回收 = (本金+利息) × 回收率 | 收益最新月 或 98% |
| 预期回款/收益（Vintage） | 预期回收率 = survival×98% + (1-survival)×30% | 假设参数 |
